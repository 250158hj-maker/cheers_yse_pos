## 1. システムの目的

### 1.1 背景（課題）

飲食店をはじめとする小規模事業者において、サービス提供時の混雑や長い待ち時間が顧客満足度の低下・売上への悪影響を招いている。また、手計算での集計は会計ミスや金銭不正のリスクがあり、業務の非効率につながっている。

### 1.2 目的

以下の3点を達成するWebレジシステムを構築することで、会計時の混雑・待ち時間の解消とスムーズな接客を実現することでリピーターを獲得する。

- **低コスト・簡単導入：** 初期設置費用を安く抑え、POSレジ未経験の個人事業者でもすぐに導入できる
- **購買動向の把握：** 販売実績（いつ・どの商品が・いくつ売れたか）から売れ筋商品を把握することができる
- **業務効率化・不正防止：** 会計ミスや金銭不正を未然に防ぎ、スムーズな精算を実現する

### 1.3 真の目的

卒業制作を見据えたチーム開発のフローを体験することで、チーム全体のスキル向上を図る。

### 1.4 ターゲットユーザー

- 設備投資にお金がかけられず、手計算もしくは安価なキャッシュレジスターで集計している小規模事業者
- POSシステムに馴染みが薄く、今までにPOSレジを使用したことがない個人事業者

### 1.5 プロダクト完成とプロジェクト成功の定義

- **完成の定義** :
    1. 「スタッフ用レジ画面（電卓UI）」と「管理者用画面」が完全に分離されて構築されていること。
    2. 管理画面から商品の登録・編集・削除ができ、それがレジ画面のボタンとして反映されること。
    3. レジ画面で計上した売上データが記録され、管理画面から「カテゴリ別」にフィルタリング表示できること。
- **成功の定義 :**
    - 小規模店舗の会計時間が半分に短縮されること。50％減が限界。
    - 店舗管理者が、日々の売上から「どのカテゴリが売れているか」を直感的に把握できるようになっていること。

## 2. アクター

| アクター | 説明 | 認証 |
| --- | --- | --- |
| **購入者（お客さん）** | 店舗の商品・サービスを購入する。システムを直接操作しない | 不要 |
| **スタッフ（店舗スタッフ）** | レジ操作・精算処理を行う。店舗IDでログインし、売上データに店舗情報が付加される | 必要（店舗IDでログイン） |
| **管理者（店舗管理者）** | 商品登録・売上データ閲覧を行う。アカウントは1つに固定 | 必要（ログイン） |

## 3. 画面構成・遷移

```markdown
ログイン画面（URLを知っていれば誰でもアクセス可能）
    ├── [レジ画面へ] → レジ画面（スタッフ用）
    └── [管理画面へ] → 管理画面（管理者用）
				    ├── 商品設定
				    └── 売上管理
```

> **前提：** ローカルネットワーク内での使用を想定しているため、URLを知らない第三者がアクセスできない環境である。
> 

> **店舗ログインについて：** 店舗ごとに1つのIDとパスワードを登録し、そのIDに店舗名などの情報を紐付ける。スタッフは自分の店舗のIDでログインするだけで店舗情報が自動的にセッションに付加されるため、レジ操作時に店舗を選択し直す必要はない。
> 

> **認証情報のフォーマット：** ログインID・パスワードはともに**数字4桁**で統一する。ローカルネットワーク内利用を前提に、スタッフが素早く打鍵できる短い数値に揃える。
> 

> [!INFO]
> **2026-05-22**: 認証情報のフォーマット（数字4桁）をチームで合意し追記。設計書に明文化されていなかったことでログイン画面のUIバリデーション仕様が宙に浮いていたため、要件レベルで確定させた。

## 4. 機能要件

### 4.1 レジ画面（スタッフ用）

### 4.1.1 電卓操作機能

| 機能 | 詳細 |
| --- | --- |
| **数字入力** | 0〜9のボタンで商品金額を入力し、画面の金額表示欄に表示する |
| **同品計算（×）** | 金額入力後に × ボタンを押し、個数を入力することで「金額 × 個数」を計算する |
| **合計計算（＋ ＝）** | ＋ボタンで次の商品金額を加算し、＝ボタンで合計を表示する |
| **消費税（8%）** | 表示中の金額に消費税8%を加算して表示する |
| **消費税（10%）** | 表示中の金額に消費税10%を加算して表示する |
| **マイナス（−）** | 表示中の金額から次に入力した金額を減算する（値引き・返品対応） |
| **リセット（AC）** | 入力中のすべての値をクリアし、金額表示欄を0に戻す |

### 4.1.2 商品ボタン機能

- 管理者が登録した商品をカテゴリ別にボタンとして表示する
- 商品ボタンを押すと、その商品の金額が金額入力欄に反映される（ショートカットキーとして機能）
- カテゴリ別にすべての商品ボタンを表示する（Airレジに近いUI）

### 4.1.3 計上処理

- 金額表示欄に合計金額が表示されている状態で「計上」ボタンを押すと、売上情報（計上日時・計上額など）をDBに保存する

### 4.1.4 売上表示処理

- 「売上」ボタンを押すと、累計の売上合計を画面に表示する

---

### 4.2 管理画面（管理者用）

### 4.2.1 認証機能

- ホーム画面の「管理画面へ」ボタンからログイン画面へ遷移する
- 管理者アカウントは1つに固定（複数アカウントへの対応はしない）
- ログイン成功後に管理画面へ遷移する

### 4.2.2 商品登録機能

管理者が商品マスタを登録・管理できるフォームを用意する。

| 項目 | 内容 |
| --- | --- |
| 商品名 | テキスト入力 |
| 金額 | 数値入力（円） |
| カテゴリ | テキスト入力（新規カテゴリの作成も可能） |
- 登録した商品はレジ画面の商品ボタンに反映される
- 登録済み商品の編集・削除も行える（詳細は詳細設計で定義）

### 4.2.3 売上データ閲覧機能

- 計上された売上データを一覧で確認できる
- 店舗別にフィルタリングして表示できる
- 表示粒度は最低限とする（詳細は詳細設計で定義）

## 5. 非機能要件

| 項目 | 内容 | 根拠 |
| --- | --- | --- |
| **動作環境（クライアント）** | Google Chrome（最新版） | 仕様書の構成一覧に準拠 |
| **動作環境（サーバー）** | Apache HTTP Server 2.4系 + MySQL 8.x | 仕様書の構成一覧に準拠 |
| **ネットワーク** | ローカルネットワーク内での使用を想定 | セキュリティ上の代替手段として機能 |
| **認証方式** | 管理者アカウント1つを固定（セッション管理） | 複数アカウント対応は開発期間1ヶ月の制約内でスコープ外 |
| **消費税率** | 8%・10%の2種類に対応 | 軽減税率への対応が必要なため |
| **データ保持** | 計上データはDBに永続保存する | 売上分析の基礎データとなるため |
| **エラーハンドリング** | 不正入力（マイナス値・過大な桁数など）に対する処理を定義する | 詳細は詳細設計で定義 |
| 可用性 | アプリケーションはいつ使えるのか・レスポンス | お会計業務に関わる時間を半分にしたいという要件から |

## 6. スコープ外（今回対応しない機能）

| 項目 | 理由 |
| --- | --- |
| 複数管理者アカウントの管理 | 開発期間1ヶ月の制約内では優先度が低い |
| 在庫管理機能 | 要件として定義されていない |
| レシート印刷 | 要件として定義されていない |
| レジドロワーとの連携 | 要件として定義されていない |
| 売上の高度な分析・グラフ表示 | 最低限の表示にとどめる方針のため |
| スマートフォン・タブレット最適化 | PC・Webブラウザでの動作を前提とするため |
| キャッシュレス決済の連携 | 開発期間の制約で困難と判断 |

## 7. 暫定DBテーブル構成

詳細はER図・テーブル定義（詳細設計書）で定義する。現時点での想定テーブルは以下の通り。

| テーブル名 | 主な項目 | 用途 |
| --- | --- | --- |
| `stores` | 店舗ID・パスワード（ハッシュ）・店舗名など | 店舗情報・スタッフ認証 |
| `products` | 商品名・金額・カテゴリ | 商品マスタ |
| `sales` | 計上日時・計上額・税率・店舗ID | 売上記録（店舗情報を付加） |
| `admin` | ログインID・パスワード（ハッシュ） | 管理者認証 |

## 9. 開発体制・スケジュール

| 項目 | 内容 |
| --- | --- |
| 開発期間 | 1ヶ月（4週間） |
| チーム人数 | 3名 |
| 役割分担方針 | 機能単位オーナー制。1機能をフロントからDB操作まで一気通貫で担当し、全員がフロント・バックエンド両方を経験する |

### 役割分担（暫定案）

| 担当 | 機能 | フロント側 | バックエンド側 |
| --- | --- | --- | --- |
| Aさん | レジ画面（計算ロジック） | 電卓UI・商品ボタン表示 | 商品一覧取得・計上API |
| Bさん | 商品登録（管理者機能） | 登録フォームUI | 商品CRUD・DBテーブル設計 |
| Cさん | 認証・売上表示 | ログイン画面・売上一覧UI | 認証処理・売上集計 |

> **注意：** BさんがDBのテーブル定義とAPIインターフェースを最初に決めることで、AさんとCさんが並行して開発できる状態を作る。
