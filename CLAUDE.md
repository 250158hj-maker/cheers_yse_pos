## ディレクトリ構成の原則

```
cheers_yse_pos/
├── public/      # 外部からアクセス可能な唯一の公開ディレクトリ
├── src/         # ビジネスロジック（外部から直接アクセス不可）
├── views/       # HTMLテンプレート（外部から直接アクセス不可）
├── config/      # DB接続情報など
└── .env         # 環境変数（Git管理外・絶対にコミットしない）
```

- `public/` 以外に外部からアクセスできる状態を作ってはならない
- DB操作・認証処理などのビジネスロジックは必ず `src/` に置く。`public/` のファイルに直接書かない
- HTMLの生成は `views/` に分離する。`public/` のPHPファイルにHTMLを直書きしない

## コード品質の原則

### データ管理

- 同じデータを複数箇所で管理しない（Single Source of Truth）
- 導出できる値は必ず導出する。二重管理はバグの温床になる

### 責任の分離

- UIと業務ロジックを同じファイル・関数に混在させない
- 1つの関数は1つのことだけ行う
- 変数・関数名は意図が伝わる名前にする（`$data`、`$tmp`、`$flag` などは使わない）

### エラーハンドリング

- `catch` の中を空にしない
- エラーをコンソール出力だけで握りつぶさない

### コメント

コードを読めばわかることは書かない。**なぜそう実装したか（理由・背景）**を書く。

```php
// NG: コードそのままの説明
// $price を 0 にセット
$price = 0;

// OK: 判断の根拠を残す
// 金額は円単位の整数で管理。小数点以下はアプリ層で丸めてから保存し、計算誤差の蓄積を防ぐ
$price = 0;
```

## PHPセキュリティルール

- ユーザー入力は必ず **バリデーション → サニタイズ → エスケープ** の順で処理する
- SQLは必ず **PDOのプリペアドステートメント**を使う。文字列結合でSQLを組み立てない
- セッション処理は `src/Auth.php` の関数を使う。各ファイルで直接 `$_SESSION` を操作しない

### 認証ガード

- `public/register/` 以下：スタッフセッションがなければ `index.php` にリダイレクト
- `public/admin/` 以下：管理者セッションがなければ `index.php` にリダイレクト
- リダイレクト処理は `src/Auth.php` に実装し、各ページで使い回す

### DB操作

- DB接続は `src/Database.php` のPDOラッパーを通して行う。直接 `new PDO()` を呼ばない
- トランザクションが必要な処理（計上処理など）は必ずトランザクションで囲む

## スコープ外（実装しない機能）

設計書で明示的にスコープ外とされている以下の機能は実装しない。

- 複数管理者アカウントの管理
- 在庫管理機能
- レシート印刷
- レジドロワーとの連携
- 売上の高度な分析・グラフ表示
- スマートフォン・タブレット最適化
- キャッシュレス決済の連携

## チーム開発ルール

- **APIインターフェース（テーブル定義）はBさんが先に決める**。AさんとCさんはそれを待ってから並行開発を始める
- 他のメンバーの担当ファイルを変更する場合は、先にSlackで確認する
- 共通ファイル（`src/Database.php`、`views/layout/` など）を変更する場合は全員に通知する

---

## 禁止事項

| 禁止 | 理由 |
|---|---|
| 設計書にない機能の追加 | スコープ管理・チーム合意の破壊につながる |
| 環境変数のハードコード | 認証情報・接続情報の漏洩リスク |
| デバッグ出力（`var_dump` など）のコミット残し | 本番環境への混入リスク |
| 複数機能の同時実装 | レビュー・デバッグが困難になる |
| 自己判断での設計変更 | 決定の記録が失われ、チームの認識がずれる |
| SQLの文字列結合 | SQLインジェクションのリスク |
